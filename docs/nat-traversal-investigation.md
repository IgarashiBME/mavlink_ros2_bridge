# NAT越え通信問題の調査記録

**日付:** 2025-03-10
**ブランチ:** `fix/nat-traversal`

## 背景

ローカルネットワークおよびOpenVPN経由ではQGCとの通信が正常に動作していたが、
ローバー側のLTEルーターでポートフォワーディングし、QGC側からグローバルIPを指定して接続する構成に変更したところ、以下の問題が発生した。

- QGC側でパラメータが受信できない
- QGC側からウェイポイントを送信できない

## ネットワーク構成の比較

### 以前（動作OK）

```
QGC <---> [同一LAN / OpenVPN] <---> Bridge:14551
          NAT なし、直接到達可能
```

### 現在（動作NG）

```
QGC:14550
  |
  v
[QGC側ルーター NAT] ← ここが問題
  | ソースポート書き換え (14550 → 10955)
  v
Internet
  |
  v
[LTEルーター] ポートフォワード: 外部:14551 → 内部:14551
  |
  v
Bridge:14551
```

## 原因

### UDPソケットのアドレス自動学習メカニズム

`udp_socket.cpp` の `receive()` は `recvfrom()` で受信パケットの送信元アドレスを
`remote_addr_` に記録し、以降の `send()` はそのアドレスへ送信する。

### ダブルNAT問題

診断ログにより、QGC側もNATの内側にいることが判明した。

```
[mavlink_bridge_node]: Remote endpoint updated:  → 126.166.217.214:10955
                                                                    ^^^^^
                                                    14550ではない = QGC側NATがポートを書き換えている
```

#### QGC → Bridge（成功）

```
QGC:14550
  → QGC側NAT (ソースポート書き換え: 14550 → 10955)
  → 126.166.217.214:10955 → Internet
  → LTEルーター:14551 (ポートフォワード DNAT)
  → Bridge:14551  ✓ 到達
```

#### Bridge → QGC（失敗）

```
Bridge:14551 → 126.166.217.214:10955 に返信
  → LTEルーターの送信NAT (ソースポート書き換え: 14551 → ????)
  → LTE_GLOBAL:???? → Internet
  → 126.166.217.214:10955 に到着
      ↑
  QGC側NATのコネクション追跡:
    期待: LTE_GLOBAL:14551 → 126.166.217.214:10955
    実際: LTE_GLOBAL:????  → 126.166.217.214:10955
    → ソースポート不一致により破棄  ✗
```

**要約:** QGC側のNATは、LTEルーターのポート14551へ送信した通信に対して
ポート14551からの応答のみを許可する（コネクション追跡）。しかしLTEルーターの
送信NATがソースポートを書き換えるため、QGC側NATが応答パケットを正規のものと
認識できず破棄する。

## コード修正内容（fix/nat-traversal ブランチ）

### 1. NAT診断ログの追加

- `UDPSocket::remote_endpoint()` メソッドを追加（送信先の IP:port を文字列で取得）
- 受信時にリモートアドレスが変化した場合にログ出力
- これにより、NATによるアドレス・ポート変換の状況を即座に確認可能

### 2. パラメータ送信のペーシング

- 変更前: `PARAM_REQUEST_LIST` 受信時に25個のパラメータを遅延なしで一括送信
- 変更後: 50ms間隔のタイマーで1つずつ送信
- 理由: LTE回線（RTT 50-200ms）ではバースト送信によるパケットロスが発生しやすい

## 解決策

### 案1: QGC側でUDPポートフォワーディング（推奨）

QGC側のルーターで以下のポートフォワーディングを設定する:

```
UDP 14550 外部 → QGC PC:14550 内部
```

これにより、ブリッジからの応答がNATのコネクション追跡に依存せず、
直接QGCに到達できるようになる。

### 案2: QGC側ルーターのNATタイプを変更

QGC側ルーターが「フルコーンNAT」をサポートしている場合、
ソースポートが異なる応答パケットも通過させることができる。
ルーターの管理画面で「NAT タイプ」や「UDPフィルタリング」の設定を確認する。

### 案3: LTE回線上でVPNを使用

OpenVPN等のVPNをLTE回線上で構築すれば、NATを完全にバイパスできる。
以前のOpenVPN構成で動作していたため、最も確実な方法。

```
QGC <--[VPN tunnel over LTE]--> Bridge
       NAT を完全にバイパス
```

### 案4: LTEルーターでソースポート保持を設定

LTEルーターの送信NATがソースポート14551を保持するよう設定できれば、
QGC側NATのコネクション追跡と一致するようになる。
ただし、LTEルーターの機種によっては設定不可の場合がある。
